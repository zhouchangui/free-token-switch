# 开发工作流

选择任务处理方式、是否需要澄清、是否要使用 worktree/分支隔离、以及如何验证业务行为时，加载本文件。

## 任务分类

- 先判断任务是简单、复杂，还是需求不明确。
- 简单任务范围窄、期望行为清楚、触及区域小，并且能用聚焦测试或检查验证。
- 复杂任务通常涉及不清晰需求、跨模块行为、高风险用户流程、数据迁移、代理/session/provider 逻辑，或多步骤 UI/后端协作。
- 如果复杂度拿不准，先询问用户，再选择工作流。

## 复杂或不明确任务

需求不清或任务复杂时使用这条链路：

- `brainstorming`：澄清意图、约束、备选方案与设计。
- `writing-plans`：在写代码前产出书面实现计划。
- `executing-plans`：按计划带检查点执行。
- `verification-before-completion`：完成前先验证，再声明结果。

开始实现前，询问是否使用 `using-git-worktrees` 创建隔离工作区，或是否创建/切换到功能分支。

## 简单且明确任务

任务窄且清楚时使用这条链路：

- `karpathy-guidelines`：显式说明假设，保持改动外科手术式，并定义可验证成功标准。
- `test-driven-development`：功能、Bug 修复、重构或行为变更先写失败测试，除非用户明确允许例外。
- `verification-before-completion`：声明完成或通过前，运行新鲜验证。

纯文档或配置改动不适合 TDD 时，保持同样纪律：做最小有效修改，然后通过阅读变更文件、检查行数、检查链接/路径等轻量方式验证。

## 日志驱动验证

复杂业务逻辑中，日志是验证设计的一部分，不是事后补充。

- 实现前或实现过程中设计预期日志信号。
- 执行业务流程，检查成功路径日志是否符合预期。
- 执行或模拟错误路径，检查错误路径日志是否覆盖预期。
- 确认日志不会暴露秘密、凭据、认证码或含密 URL。
- 优先用自动化触发业务流程：单元测试、集成测试、命令调用、脚本化 UI 操作或本地浏览器自动化。
- 自动化无法覆盖时，向用户说明需要的人工步骤和具体证据。

## UI 流程验证工具

- 适合浏览器自动化时，对 localhost、file URL 或内置浏览器检查使用 `browser-use:browser`。
- 桌面应用交互无法通过浏览器自动化完成时，使用 `@Computer Use`。
- UI 相关代码变更后，使用 DOM 状态、截图、console 日志、应用日志或 emitted events 等具体现象验证，不只凭视觉印象。
