# 如何开发兼容 clawtip 支付体系的 Skill

## 📖 概述

本文档旨在指导开发者编写能够兼容 `clawtip` 支付流程的 开发者服务Skill。

![开发者技能支付技能交互逻辑.jpg](%E5%BC%80%E5%8F%91%E8%80%85%E6%8A%80%E8%83%BD%E6%94%AF%E4%BB%98%E6%8A%80%E8%83%BD%E4%BA%A4%E4%BA%92%E9%80%BB%E8%BE%91.jpg)
说明了您的开发者服务Skill与`clawtip`支付Skill进行交互的逻辑。在一次交互中，需要您首先创建对用户的请求创建其对应的订单。
然后将订单信息交由`clawtip`技能实现从用户钱包到您钱包的交易，在交易后，支付技能会将`支付凭证`写入订单文件。
再之后，其包含了用户的交易结果等信息。您需要从订单文件中读取`支付凭证`并提交给服务端进行校验以获取交易结果，并根据交易结果决定是否对用户进行服务。

因而一个标准的开发者服务Skill由三个阶段组成：

- **Phase 1 — 创建订单**：调用你自己的脚本，生成订单信息，并将订单数据保存到本地固定目录的 JSON 文件中
- **Phase 2 — 支付处理**：将 `order_no` 和 `indicator` 传给 `clawtip` 完成支付，支付凭证会被自动写入订单文件
- **Phase 3 — 执行服务**：传入 `order_no`，脚本自动从订单文件读取`支付凭证`和业务参数，调用你的服务后端验证并提供服务

---

## 📁 推荐的开发者服务Skill项目结构

    your-skill-name/
        ├── SKILL.md               # Skill 主描述文件（含工作流说明）
        └── scripts/
            ├── create_order.py    # Phase 1：创建订单脚本
            ├── your_service.py    # Phase 3：业务执行脚本
            └── file_utils.py      # 订单文件读写工具

---
**注意** 此处未包含服务端部分，`create_order.py` 以及 `your_service.py` 需要通过向后端发起对应的 API 请求来完成订单创建和业务执行。

## 📂 订单文件机制

新版本引入了**本地订单文件机制**，用于在各阶段之间传递订单数据和支付凭证。

### 工作原理

1. **Phase 1**：`create_order.py` 创建订单后，将订单详细信息（`order_no`、`amount`、`encrypted_data`、`pay_to`、`description`、`slug`、`resource_url` 等）保存到固定目录下的 JSON 文件。
2. **Phase 2**：`clawtip` 支付技能根据 `order_no` 和 `indicator` 自动定位并读取该 JSON 文件中的支付参数，支付完成后将 `payCredential`（支付凭证）写回同一文件。
3. **Phase 3**：`your_service.py` 根据 `order_no` 从订单文件中读取 `question`（业务参数）和 `payCredential`（支付凭证），调用服务端接口。

### 文件存储路径

- **Linux/macOS**: `~/.openclaw/skills/orders/{indicator}/{order_no}.json`
- **Windows**: `~/openclaw/skills/orders/{indicator}/{order_no}.json`

### indicator 的计算

`indicator` 是通过对技能的 `slug`（技能唯一标识符）进行 MD5 哈希计算得出的，用于区分不同技能的订单目录。

```python
import hashlib
indicator = hashlib.md5("your-skill-slug".encode("utf-8")).hexdigest()
```

### file_utils.py 工具模块

我们提供了 `file_utils.py` 工具模块用于订单文件的读写操作，开发者应在 `create_order.py` 和 `your_service.py` 中引用该模块。请参考 [file_utils.py](Skill%20%E6%A8%A1%E6%9D%BF%2F%E5%AE%8C%E5%85%A8Skill%E6%A8%A1%E6%9D%BF%2Fscripts%2Ffile_utils.py)。

## ⚙️ 脚本开发规范

### 创建订单脚本：`create_order.py`
**基础说明**
`create_order.py` 是用于创建订单的脚本。它需要用户输入的问题（`question`），向服务端发起创建订单请求，将订单数据保存到本地 JSON 文件，并返回订单号（`ORDER_NO`）、订单金额（`AMOUNT`）、用户问题（`QUESTION`）和指示器（`INDICATOR`）。

**详细说明（建议开发者按此理解实现）**

`create_order.py` 的职责是"参数采集 + HTTP 转发 + 订单文件落盘 + 关键字段透传"，而不是在脚本里实现完整业务后端逻辑。推荐流程如下：

1. 脚本从命令行参数中读取用户输入（通常是 `question`，也可扩展附加参数）。
2. 脚本根据硬编码的 `slug` 计算 `indicator`（MD5 哈希）。
3. 脚本将这些参数通过 HTTP 请求发送到你自己的开发者服务端"创建订单接口"。
4. 服务端完成真正的业务逻辑（例如生成订单号、计算金额、组装并加密 `orderNo/amount/payTo`）。
5. 脚本从服务端响应中提取关键字段，**将完整订单数据（包括 `encrypted_data`、`pay_to`、`description`、`slug`、`resource_url` 等）保存到本地 JSON 文件**。
6. 脚本按照本文固定格式 `print` 关键字段到控制台，供 openclaw 客户端读取。

服务端逻辑建议参考[服务端开发模板.md](%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%BC%80%E5%8F%91%E6%A8%A1%E6%9D%BF%2F%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%BC%80%E5%8F%91%E6%A8%A1%E6%9D%BF.md)中：
`public Map<String, Object> createOrder(@RequestBody Map<String, String> request)`  
这部分方法是"订单创建"的主处理入口，脚本只负责调用它并转发结果，不要把核心下单逻辑堆在脚本里。

**为什么必须打印固定字段**

openclaw 客户端会从脚本标准输出中读取 `ORDER_NO`、`AMOUNT`、`QUESTION`、`INDICATOR`。  
如果字段名、分隔符、大小写或输出顺序被随意改动，会导致客户端解析失败，进而影响支付流程。

**为什么必须保存订单文件**

新版本中，`clawtip` 支付技能不再接收 `encrypted_data`、`pay_to` 等参数，而是根据 `order_no` 和 `indicator` 从本地文件中自动读取。因此，`create_order.py` **必须**在输出前将完整订单数据保存到固定目录。

**实现边界建议**

- 脚本层：参数校验、HTTP 调用、异常捕获、订单文件保存、固定格式输出。
- 服务端层：业务校验、订单落库、金额计算、SM4 加密、风险控制、审计日志。
- 不限行业：即使你不是天气类服务，也沿用同样流程；只需把服务端的业务内容替换为你的实际服务。

**必须遵守的输出格式（成功时）：**

    ORDER_NO=<value>
    AMOUNT=<value>
    QUESTION=<value>
    INDICATOR=<value>

    请注意，`ORDER_NO` 代表您系统中的内部订单号，`AMOUNT` 代表订单金额（以分为单位），`QUESTION` 代表用户的查询内容，`INDICATOR` 是您技能 slug 的 MD5 哈希值。

**💡 关于订单 JSON 文件的说明：**

`create_order.py` 保存的 JSON 文件应包含以下字段，供 `clawtip` 支付技能和 Phase 3 脚本读取：

```json
{
  "skill-id": "your-skill-id",
  "order_no": "订单号",
  "amount": "金额（分）",
  "question": "用户问题",
  "encrypted_data": "SM4加密后的订单JSON",
  "pay_to": "收款钱包地址",
  "description": "服务描述",
  "slug": "技能slug标识",
  "resource_url": "服务资源地址"
}
```

其中 `encrypted_data` 是使用开发者的 SM4 对称密钥（即您在邮件里收到的安全密钥）对包含 `orderNo`、`amount`、`payTo` 的 JSON 串加密的结果。

服务端加密/解密算法请参考文件[服务端SM4加解密代码示例.md](%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%BC%80%E5%8F%91%E6%A8%A1%E6%9D%BF%2F%E6%9C%8D%E5%8A%A1%E7%AB%AFSM4%E5%8A%A0%E8%A7%A3%E5%AF%86%E4%BB%A3%E7%A0%81%E7%A4%BA%E4%BE%8B.md)
**请勿在 scripts 脚本目录内直接执行服务端逻辑。**

**💡 关于 `CREDENTIAL`(支付平台支付凭证) 的说明：**

支付成功后，`clawtip` 支付技能会将 `payCredential` 自动写入订单 JSON 文件中。Phase 3 脚本只需根据 `order_no` 从文件中读取即可。

**`CREDENTIAL`解密后为 JSON 结构【clawtip返回的credential数据】：**

解密后得到的字典一般包含以下五个字段（均为 `string` 类型）：

| 字段 | 类型 | 说明                            |
|------|------|-------------------------------|
| `orderNo` | string | 订单号                           |
| `amount` | string | 金额                            |
| `payTo` | string | 收款地址                          |
| `payStatus` | string | 支付状态(SUCCESS、FAIL、PROCESSING) |
| `finishTime` | string | 完成时间(yyyy-MM-dd HH:mm:ss) |

示例：
```json
{
  "orderNo": "202603280001",
  "amount": "100",
  "payTo": "0x1234...abcd",
  "payStatus": "SUCCESS",
  "finishTime": "2026-03-31 12:00:00"
}
```

**必须遵守的输出格式（失败时）：**
当订单创建失败时，必须按照下列格式将失败提示输出到控制端，告知用户

    订单创建失败: <error detail>

并以 `exit(1)` 退出。

`create_order.py`创建订单脚本可以参考我们提供的[create_order.py](Skill%20%E6%A8%A1%E6%9D%BF%2F%E5%AE%8C%E5%85%A8Skill%E6%A8%A1%E6%9D%BF%2Fscripts%2Fcreate_order.py)

### 发起支付 
您向 `clawtip` 技能传入的参数请参照以下部分，进行严格约定。

| 字段名称 | 类型 | 必填 | 说明 | 示例 |
|---|---|---|---|---|
| `order_no` | `string` | 必填 | 交易单号，≤32字符 | 202603021109... |
| `indicator` | `string` | 必填 | 技能 slug 的 MD5 哈希值 | a1b2c3d4e5f6... |

> **注意：** 新版本中，`clawtip` 支付技能会根据 `order_no` 和 `indicator` 自动从本地订单 JSON 文件中读取 `pay_to`、`encrypted_data`、`amount`、`question`、`description`、`slug`、`resource_url` 等参数。这些参数由 Phase 1 的 `create_order.py` 写入文件，无需在调用时手动传递。

订单 JSON 文件中应包含以下字段（由 Phase 1 写入）：

| 字段名称 | 类型 | 说明                        | 示例                    |
|---|---|---------------------------|-----------------------|
| `skill-id` | `string` | 技能ID                      | si-weather-reporter   |
| `order_no` | `string` | 交易单号                      | 202603021109...       |
| `amount` | `long` | 金额（单位：分）                  | 1（0.01元）              |
| `question` | `string` | 用户请求内容                    | 我要查天气                 |
| `encrypted_data` | `string` | 国密加密的订单JSON（SM4加密后的JSON串） | Sm2Encrypted...       |
| `pay_to` | `string` | 收款服务ID（邮件获取）              | 0x742d...             |
| `description` | `string` | 交易描述（选填），≤128字符           | 天气信息服务                |
| `slug` | `string` | 技能唯一标识符                   | clawtip-weather       |
| `resource_url` | `string` | 服务提供地址，≤128字符             | `https://ms.jr.jd.com` |



### 提供服务脚本：`your_service.py`

**基础说明**

`your_service.py` 是"履约查询/服务发放"阶段脚本，用于接收 `<order_no>`，从订单文件中自动读取 `question` 和 `payCredential`，调用你自己的服务端查询接口，并把履约状态按约定输出给 openclaw 客户端。

**详细说明（建议开发者按此理解实现）**

推荐流程如下：

1. 脚本读取命令行参数：订单号 `order_no`。
2. 脚本根据硬编码的 `slug` 计算 `indicator`，然后从本地订单文件中读取 `question`（业务参数）和 `payCredential`（支付凭证）。
3. 脚本通过 HTTP 调用开发者服务端的"结果查询/履约接口"（通常对应后端 `getServiceResult` 或同等业务方法），传递 `question`、`order_no`、`credential`。
4. 服务端执行核心逻辑：解密 `credential`、判断 `payStatus`、校验订单、判断是否已履约、必要时执行一次履约并落库履约记录。
5. 脚本将服务端结果中的关键字段打印到控制台，供 openclaw 客户端消费。

服务端逻辑建议参考[服务端开发模板.md](%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%BC%80%E5%8F%91%E6%A8%A1%E6%9D%BF%2F%E6%9C%8D%E5%8A%A1%E7%AB%AF%E5%BC%80%E5%8F%91%E6%A8%A1%E6%9D%BF.md)中：
    `public Map<String, Object> getServiceResult(@RequestBody Map<String, String> request)`
这部分方法是"履约查询/服务发放"的主处理入口，脚本只负责调用它并转发结果，不要把核心下单逻辑堆在脚本里。
关键点：脚本层不要替代服务端做支付凭证可信判断。  
支付状态判断与幂等履约控制应在服务端完成，脚本只做转发和标准化输出。
商家需依据自身业务逻辑对交易凭证的有效性进行校验，以防范重复提供服务所导致的资金损失。

**输出约定与语义建议**

- 当服务端确认可履约或已成功返回业务结果时，输出 `PAY_STATUS: SUCCESS`。
- 当服务端仍在处理中时，输出 `PAY_STATUS: PROCESSING`。
- 当支付失败或业务失败时，输出 `PAY_STATUS: FAIL`。
- 当参数错误、解密失败、系统异常等不可继续场景时，输出 `PAY_STATUS: ERROR`，并补充 `ERROR_INFO: <原因>`。

**业务通用性说明**

但该脚本并不限定天气业务。  
任何"支付后提供一次服务结果"的场景（如报告生成、数据查询、权益发放）都可复用同一接入模式。

**必须遵守的输出字段：**

| 字段 | 枚举                                    |输出格式(您向控制台输出的格式)                                                                        | 是否必须 |
|------|---------------------------------------|-----------------------------------------------------------------------------------------|----------|
| `PAY_STATUS` | `SUCCESS`,`PROCESSING`,`FAIL`,`ERROR` | `PAY_STATUS: SUCCESS`        | ✅ 必须 |
|  `ERROR_INFO`| N/A                                   | `ERROR_INFO: <错误原因>`          | ✅ 当 PAY_STATUS 为 ERROR 时必须 |

`your_service.py` 可参考 [weather_report.py](Skill%20%E6%A8%A1%E6%9D%BF%2F%E5%AE%8C%E5%85%A8Skill%E6%A8%A1%E6%9D%BF%2Fscripts%2Fweather_report.py)

## 🧪 沙箱测试环境

在正式发布技能之前，我们强烈建议您使用 **clawtip-sandbox（沙箱环境）** 对技能进行快速测试和验证。沙箱环境**不涉及真实资金交易**，无需钱包注册，专为开发者调试而设计。

### 如何启用沙箱

只需在您的技能 SKILL.md 中，将 Phase 2（支付处理）的调用技能名称从 `clawtip` 替换为 `clawtip-sandbox` 即可：

**正式环境（替换前）：**
```
**操作：** 使用技能 `clawtip` 处理支付并获取支付凭证。若该技能不存在，需先安装。
```

**沙箱环境（替换后）：**
```
**操作：** 使用技能 `clawtip-sandbox` 处理支付并获取支付凭证。若该技能不存在，需先安装。
```

> **完成测试后，请务必替换回 `clawtip` 再正式发布。**

### 沙箱与正式环境的差异

| 对比项 | 正式环境 (`clawtip`) | 沙箱环境 (`clawtip-sandbox`) |
|--------|---------------------|---------------------------|
| 真实资金 | ✅ 涉及真实交易 | ❌ 不涉及真实资金 |
| 钱包注册 | ✅ 需要用户注册钱包 | ❌ 无需注册 |
| Token 加密 | ✅ 需要加密用户 Token | ❌ 使用固定测试 Token |
| 调用参数 | `order_no` + `indicator` | `order_no` + `indicator`（相同） |
| 调试输出 | 仅返回支付结果 | 返回详细调试信息，包括订单文件参数解析 |
| 错误诊断 | 通用错误提示 | 精确定位到开发者技能配置中的具体出错字段 |

### 沙箱的调试能力

沙箱会输出完整的调试信息，包括 `========== 订单文件请求参数 ==========` 参数块，展示从您订单文件中读取到的所有字段值，帮助您快速定位问题。

### 沙箱错误诊断指南

当沙箱测试出现错误时，所有错误均指向**开发者自身技能配置的问题**，而非沙箱基础设施问题。以下是常见错误类型及排查方向：

#### 1. 参数检查失败
- **表现：** 输出包含 `参数检查失败:`
- **原因：** 订单文件中缺少必填字段
- **排查：** 检查 `create_order.py` 中保存的订单 JSON 是否包含所有必填字段（`payTo`、`amount`、`order_no`、`encrypted_data`、`slug`、`question`、`description`、`resource_url`）

#### 2. 服务器请求异常
- **表现：** 输出包含 `服务器请求服务时异常`
- **原因：** 网络问题或参数格式严重异常
- **排查：** 检查网络连接，确认订单文件中的参数格式正确

#### 3. 验证未通过（NOTPASS）
- **表现：** 输出包含 `状态: NOTPASS`
- **原因：** 服务端验证失败，技能配置有误
- **常见问题及对策：**

| 错误信息关键词 | 根因 | 修复建议 |
|--------------|------|---------|
| `slug已被其他商户使用` | 技能 slug 与其他已注册技能冲突 | 更换一个唯一的 slug 值 |
| `商家信息有误` / `解密失败` | `encrypted_data` 密钥不匹配 | 确认使用了与 `payTo` 对应的正确对称密钥加密 |
| `收款地址` / `payTo` | 收款地址无效或未注册 | 确认 `payTo` 已在平台注册且格式正确 |
| `金额` / `amount` | 金额参数异常 | 确认金额为正整数（单位：分） |
| `订单` / `order` | 订单号重复或格式错误 | 确认订单号格式正确且未重复使用 |

#### 4. 支付模拟失败（TEST_FAIL）
- **表现：** 输出包含 `支付状态: TEST_FAIL`
- **原因：** 通过了基本验证但支付模拟失败，通常是 `encrypted_data` 与密钥管理问题
- **排查重点：**
  1. `encrypted_data` 是否使用了与 `payTo` 对应的正确密钥加密？
  2. `slug` 是否是您的技能独有的标识？
  3. `payTo` 收款地址是否已在平台上正确注册？

#### 5. 支付模拟成功（TEST_SUCCESS）
- **表现：** 输出包含 `支付状态: TEST_SUCCESS` 和 `状态: PASS`
- **含义：** 恭喜！您的技能配置验证通过
- **后续步骤：** 使用您的商家密钥解密返回的 `payCredential`，对比 `payCredential明文` 验证解密流程正确性。验证完毕后，将 SKILL.md 中的 `clawtip-sandbox` 替换回 `clawtip` 即可正式上线

### 沙箱测试流程总结

```
1. 在 SKILL.md Phase 2 中将 clawtip 替换为 clawtip-sandbox
2. 正常触发技能流程（Phase 1 → Phase 2 → Phase 3）
3. 观察沙箱输出的调试信息，定位并修复问题
4. 重复测试直到收到 TEST_SUCCESS
5. 将 clawtip-sandbox 替换回 clawtip，正式发布
```
