# 好友分享与卖了换钱功能完善路线图

**日期：** 2026-04-25
**目标：** 将当前“可演示”的好友分享和市场售卖能力，逐步补齐为安全、可用、可验证的产品闭环。

## 标记约定

- `[ ]` 未开始
- `[x]` 已完成
- 每个优先级建议从上到下执行
- 若某项决定暂缓，可标记为 `[~]` 并在后面补充原因

## 推荐执行顺序

1. P0：安全止血
2. P1：好友分享可用 MVP

## P0：安全止血

> 目标：先避免把本地代理裸露到公网，确保功能不会被用户误解为完整商用能力。

- [x] 给“好友分享”和“卖了换钱”增加实验性标识或功能开关。
- [x] 外部请求必须校验分享 token，token 缺失或错误时直接拒绝。
- [x] 停止好友分享时让对应 token 失效。
- [x] 停止好友分享时关闭对应 tunnel，或至少从后端运行态移除。
- [x] UI 的“运行中”状态改为以后端运行态为准，不能只读 `shareConfig` 持久化字段。
- [x] 分享启动日志、错误日志和复制信息中不得泄露原始 provider API key 或完整 provider 密钥；好友导入链接仍会携带本次分享 token，停止后失效，P1 再升级为短期凭证。
- [x] 明确当前版本限制：好友分享和市场售卖暂不保证多 provider 并行。

**2026-04-25 止血实现记录：** 当前版本已强制外部请求携带已登记分享 token；启动分享/售卖会生成新 token；停止会让旧 token 失效并停止当前单 tunnel；UI 运行态以后端 tunnel + active token 为准；界面提示该能力仍为实验性、受限能力。

### P0 验收标准

- [x] 只有 tunnel 地址但没有正确 token 的外部请求会被拒绝。
- [x] 停止分享后，旧 token 不能继续访问。
- [x] 应用重启或 tunnel 进程退出后，UI 不再显示假“运行中”。
- [x] 用户能明确看到这是实验性能力或受限能力。

## P2B：流式按次付费与单席位调度

> 目标：市场售卖采用“每次调用推理服务都付费”的流式占用模型；同一时间只服务一个买家，占用期间从市场下架或标记忙碌，释放后自动重新上架。

- [ ] 定义市场 listing 状态：`available`、`reserved`、`busy`、`cooldown`、`offline`。
- [ ] relay listing 中声明当前状态和状态更新时间。
- [ ] relay listing 中声明 `capacity=1`，明确同一时间只服务一个用户。
- [ ] relay listing 中声明是否支持流式响应。
- [ ] relay listing 中声明单次调用的支付策略：调用前预付、调用中分段扣费、调用后结算。
- [ ] 买家开始调用前，seller 端创建一次调用会话。
- [ ] 调用会话创建成功后，seller 端将 provider 状态切到 `reserved`。
- [ ] 买家完成支付校验后，seller 端将 provider 状态切到 `busy`。
- [ ] 状态进入 `busy` 后，relay 市场不再把该 provider 作为可接入资源展示。
- [ ] 如果 relay 不支持删除 listing，则发布同一 provider 的 `busy` 状态覆盖旧 listing。
- [ ] 如果买家预占但未支付，超时后释放并重新上架。
- [ ] 如果推理请求正常结束，记录用量和订单后释放占用。
- [ ] 如果推理请求失败，按失败阶段决定退款、免扣或部分扣费策略。
- [ ] 如果买家断开流式连接，立即停止上游请求或等待安全终止。
- [ ] 买家断开后记录已产生 token 用量。
- [ ] 买家断开后释放占用并重新上架。
- [ ] seller 端必须有占用锁，防止两个买家同时进入同一个 provider。
- [ ] 占用锁必须有 TTL，避免异常崩溃后永久忙碌。
- [ ] 应用重启后恢复时，默认将未完成调用标记为异常结束，并把 provider 重新上架或进入 `cooldown`。
- [ ] `stop_selling_tokens` 必须能强制释放占用并下架 listing。
- [ ] UI 展示当前市场状态：空闲可售、已被占用、等待支付、冷却中、已下架。
- [ ] UI 展示当前占用用户的脱敏标识或订单 ID。
- [ ] UI 展示当前调用已持续时长。
- [ ] UI 展示当前调用已产生 token 用量和预估费用。

### P2B 状态流转

```text
offline -> available -> reserved -> busy -> available
                         |          |
                         |          -> cooldown -> available
                         |
                         -> available
```

### P2B 调用流程

1. seller 启动市场售卖，发布 `available` listing。
2. buyer 从 relay 市场发现 `available` listing。
3. buyer 请求 seller endpoint 创建调用会话。
4. seller 获取占用锁，状态变为 `reserved`，并更新 relay listing。
5. seller 复用 P2-0 验证通过的 ClawTip 适配器，生成本次调用的支付请求。
6. buyer 完成支付。
7. seller 校验支付成功，状态变为 `busy`，并更新 relay listing。
8. seller 开始转发推理请求。
9. seller 统计流式响应中的 token 用量。
10. 推理结束后，seller 写入订单和用量。
11. seller 释放占用锁，状态回到 `available`，并重新发布 listing。

### P2B 测试清单

- [ ] 空闲 provider 会发布 `available` listing。
- [ ] 第一个买家创建会话后 listing 变为 `reserved`。
- [ ] `reserved` 状态下第二个买家会被拒绝。
- [ ] 支付成功后 listing 变为 `busy`。
- [ ] `busy` 状态下第二个买家会被拒绝。
- [ ] 推理正常结束后 listing 回到 `available`。
- [ ] 支付超时后 listing 回到 `available`。
- [ ] 流式连接断开后释放占用。
- [ ] 应用重启后不会保留永久 `busy` 状态。
- [ ] 强制停止售卖后 listing 变为 `offline`。
